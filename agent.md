# agent.md

## Mission
Помогать команде быстро и предсказуемо разрабатывать продукт по системе `REQ -> CONTEXT -> STATE -> TDD -> DESIGN`.

## Source of Truth
1. `REQ.md` - зачем и для кого делаем.
2. `CONTEXT.md` - как устроена система и какие глобальные правила.
3. `STATE.md` - что уже сделано, что в работе, что дальше.
4. `TDD.md` - как проверяем качество через тесты.
5. `DESIGN.md` - визуальная система и UI-правила.

## Routing Rules
- Любой новый запрос сначала маппится на один из документов.
- Если запрос про бизнес-цели/ценность: обновлять `REQ.md`.
- Если запрос про архитектуру/ограничения: обновлять `CONTEXT.md`.
- Если запрос про план/статус/бэклог: обновлять `STATE.md`.
- Если запрос про тесты/регрессии: обновлять `TDD.md`.
- Если запрос про UI/токены/компоненты: обновлять `DESIGN.md`.

## Working Loop
1. Прочитать `REQ.md` и `CONTEXT.md`.
2. Сверить текущую задачу с `STATE.md`.
3. Сформировать тест-кейсы по `TDD.md`.
4. Реализовать изменение.
5. Обновить `STATE.md` (результат, риски, следующие шаги).
6. При изменении UI синхронизировать `DESIGN.md`.

## Quality Gates
- Нет изменений без явного acceptance criteria в `REQ.md`.
- Нет изменений в архитектуре без фиксации решения в `CONTEXT.md`.
- Нет закрытия задачи без обновления статуса в `STATE.md`.
- Нет фичи без тест-кейсов/чеков из `TDD.md`.

## Document Hygiene
- В каждом документе держать разделы: `Last updated`, `Owner`, `Status`.
- Писать коротко и проверяемо: факты, решения, критерии.
- Не дублировать один и тот же факт в нескольких документах.
